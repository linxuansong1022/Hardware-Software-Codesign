# 人脸识别门禁系统 — 开发规格说明书

**课程**: DTU 02214 — 嵌入式AI（2026春季）
**团队**: 3人
**硬件**: XIAO ESP32S3 Sense（OV2640摄像头、8MB PSRAM、8MB Flash、板载LED）
**日期**: 2026-02

---

## 1. 应用描述

### 1.1 目的

一个完全运行在 XIAO ESP32S3 Sense 微控制器上的实时人脸识别系统，作为门禁控制原型。系统为已注册的团队成员开放权限，拒绝所有其他人。

### 1.2 范围

- **使用环境**: 仅限室内（实验室、办公室、走廊）
- **用户**: 3名注册团队成员 + 所有其他人归类为"Unknown"
- **有效距离**: 摄像头前方 30–100 cm
- **输出方式**: 板载LED反馈 + 通过USB串口传输到PC端的Python实时显示界面

### 1.3 功能需求

| 编号 | 需求 |
|:----:|:-----|
| FR-1 | 系统以 ≥ 1 FPS 的速率从OV2640摄像头采集帧 |
| FR-2 | 系统将每帧分类为4类之一：成员A、成员B、成员C、Unknown |
| FR-3 | 当以 ≥ 0.80 的置信度识别到团队成员时，板载LED点亮（绿色） |
| FR-4 | 当人脸为Unknown或置信度 < 0.80 时，LED保持熄灭 |
| FR-5 | 每帧图像和推理结果通过USB串口发送到主机PC |
| FR-6 | Python显示应用展示实时摄像头画面、识别姓名和置信度条 |

### 1.4 非功能需求

| 编号 | 指标 | 目标值 | 依据 |
|:----:|:-----|:------:|:-----|
| NFR-1 | 测试集准确率 | ≥ 90% | 4分类、自采数据、分布接近，90%是合理基线 |
| NFR-2 | 每类召回率 | ≥ 85% | 防止某个人被系统性误判 |
| NFR-3 | 推理延迟 | ≤ 200 ms | ESP32S3 @ 240 MHz 运行INT8 CNN，预计约50–100 ms |
| NFR-4 | 模型二进制大小 | ≤ 50 KB | 在8 MB Flash中轻松容纳 |
| NFR-5 | Tensor arena大小 | ≤ 100 KB | 必须在~512 KB SRAM中容纳，同时留空间给其他缓冲区 |
| NFR-6 | 端到端帧率 | ≥ 1 FPS | 门禁场景足够；非实时视频需求 |
| NFR-7 | INT8量化精度损失 | ≤ 2% 准确率下降 | 量化模型须接近浮点基线 |


## 2. 数据集规格

### 2.1 类别与标注指南

| 类别编号 | 类别名称 | 描述 |
|:-------:|:-------:|:-----|
| 0 | 成员A | 团队成员1 — 正面人脸在画面中 |
| 1 | 成员B | 团队成员2 — 正面人脸在画面中 |
| 2 | 成员C | 团队成员3 — 正面人脸在画面中 |
| 3 | Unknown | 非团队成员的人脸，或无人脸 |

**标注规则**:
- **成员X**（0/1/2）: 人脸占图像面积 ≥ 30% 且清晰可辨。包含戴眼镜、轻微侧头（< 45°）、轻度遮挡的情况。
- **Unknown**（3）: 任何其他人的脸、空背景、物体、打印照片、> 45° 侧头、或身份模糊的情况。
- **丢弃**（不保存）: 运动模糊严重无法辨认面部特征，或人脸占图像面积 < 15%。
- **边缘情况**: 画面中同时出现两人 → 标注为Unknown，除非一个团队成员明显占据画面中心。

### 2.2 条件因素

| 因素 | 变化范围 |
|:----|:---------|
| 距离 | 30 cm、60 cm、100 cm |
| 角度 | 正面（0°）、左偏30°、右偏30°、轻微俯仰 |
| 光照 | 窗户自然光、顶部荧光灯、弱光（台灯）、逆光 |
| 表情 | 中性、微笑、说话 |
| 配饰 | 无眼镜、眼镜、墨镜（部分样本） |
| 背景 | 实验室桌面、墙壁、户外 |
| Unknown变化 | 空桌面、空墙壁、户外场景、陌生人（已取得同意）、部分可见人脸、打印照片 |

### 2.3 采集策略

**目标数量**: 每类200-250张 × 4类 = **共800-1000张**

**每人采集任务**（成员自行采集，目标每人约220张）:
- 任务A: 正面、中性表情、无配饰 — 50张 × 3个距离（30/60/100 cm）
- 任务B: 侧面（±30°）、轻微俯仰、不同表情 — 40张
- 任务C: 不同光照条件（自然光、荧光灯、弱光、逆光） — 40张
- 任务D: 戴眼镜/配饰 — 30张
- 任务E: 不同背景（实验室、墙壁、户外） — 30张
- 任务F: 说话/动态表情 — 30张

**Unknown类**（目标约220张）:
- 90张（~40%）: 陌生人正面照（现场真人，取得口头同意）
- 35张（~15%）: 平板/手机屏幕显示的网络人脸照片（用于反欺骗训练）
- 35张（~15%）: 部分遮挡/画面边缘人脸
- 35张（~15%）: 空背景（桌面、墙壁、户外）
- 25张（~15%）: 非人脸物体（手、书本、杯子等，防止误触发）

**数据质量要求**:
- 各类数量差距不超过20%（最少的类不低于200张）
- 每个成员的数据必须覆盖所有条件因素组合，避免"全部正面自然光"的偏倚
- 同一次连续采集的帧不能同时出现在训练集和测试集（按采集session分割，非随机打乱）

**平衡检查**: 训练前用 `ls face/data/<class>/ | wc -l` 验证各类数量。

### 2.4 数据来源与格式

- **摄像头**: XIAO ESP32S3 Sense 上的 OV2640 — 320×240 RGB565，通过 `collect.py` 保存为PNG
- **未使用公开数据集** — 所有图像均为现场采集，确保隐私合规
- **相机设置**: 固定320×240，约1 FPS采集率，无手动对焦
- **文件命名**: `image_YYYYMMDD_HHMMSS.png`（自动生成）

### 2.5 采集日志

每次采集会话记录在 `data/session_log.csv` 中:

| 字段 | 描述 |
|:----|:-----|
| `timestamp` | 会话开始时间 |
| `class` | person_a / person_b / person_c / unknown |
| `collector` | 采集者姓名 |
| `lighting` | daylight / overhead / low / backlit |
| `accessories` | none / glasses / sunglasses |
| `background` | lab / wall / outdoors |
| `notes` | 任何偏差或特殊情况 |

### 2.6 隐私与法律合规

- **团队成员**: 3人均给予明确知情同意。图像仅存储在本地。
- **Unknown类中的陌生人**: 拍摄前取得口头同意。项目完成后立即删除。
- **屏幕显示照片**: 通过平板/手机屏幕展示网络人脸图片进行采集，仅用于反欺骗训练。
- **GDPR合规**: 数据仅用于课程教育目的。项目完成后删除所有第三方图像。

## 3. 设计空间与决策论证

本节展示关键设计维度、考虑过的替代方案以及每个选择的依据。这是工程权衡分析的核心部分。

### 3.1 输入分辨率

| 选项 | 模型输入大小 | Tensor Arena（估计） | 优点 | 缺点 |
|:----|:----------:|:------------------:|:-----|:-----|
| 32×32 | 1,024 B | ~40 KB | 最小模型，最快推理 | 人脸细节丢失多，可能欠拟合 |
| **48×48** | 2,304 B | ~80 KB | 细节与资源的良好平衡 | 模型大小适中 |
| 64×64 | 4,096 B | ~150 KB | 最好的细节保留 | Arena可能超出SRAM预算；推理更慢 |

**选择: 48×48**。32×32下难以可靠区分3个外貌相似的人。64×64下tensor arena接近SRAM上限，几乎没有空间留给图像缓冲区。48×48在100 KB arena预算内提供了足够的空间细节用于4分类任务。

### 3.2 颜色通道

| 选项 | 输入形状 | 第一层计算量 | 优点 | 缺点 |
|:----|:-------:|:----------:|:-----|:-----|
| **灰度** | (48, 48, 1) | 1× | 更小模型，更简单预处理 | 丢失肤色信息 |
| RGB | (48, 48, 3) | 3× | 更丰富的特征 | 第一层计算量3倍；颜色随光照变化大 |

**选择: 灰度**。人脸身份主要依赖几何结构（眼睛、鼻子、嘴巴间距），而非颜色。室内光照变化使颜色不可靠。灰度将第一层计算量减少3倍，并简化了ESP32的预处理流程（从RGB565单通道转换）。

### 3.3 模型架构深度

| 选项 | 参数量（估计） | 模型大小（INT8） | Arena（估计） | 备注 |
|:----|:----------:|:--------------:|:-----------:|:-----|
| 2层Conv | ~8,000 | ~15 KB | ~50 KB | 4类人脸任务可能欠拟合 |
| **3层Conv** | ~22,400 | ~40 KB | ~80 KB | 容量充足；符合资源预算 |
| 4层Conv | ~50,000 | ~70 KB | ~130 KB | Arena接近SRAM上限 |
| MobileNet风格 | ~30,000 | ~45 KB | ~90 KB | DepthwiseConv2D可能缺少TFLite Micro算子支持 |

**选择: 3层Conv2D + Dense**。2层有欠拟合4个不同身份的风险。4层将arena推到100 KB以上。MobileNet风格的深度可分离卷积可以减少参数量，但 `DepthwiseConv2D` 需要向TFLite Micro的算子解析器添加新算子——在受限平台上有风险。3层架构复用了keywords项目中已验证的完全相同的算子集（`Conv2D`、`MaxPool2D`、`FullyConnected`、`Softmax`）。

### 3.4 量化策略

| 选项 | 模型大小 | 推理速度 | 精度影响 |
|:----|:-------:|:-------:|:---------|
| Float32（不量化） | ~90 KB | 较慢（整数核上做浮点运算） | 基线 |
| **Full INT8** | ~40 KB | 最快（ESP32S3原生int8） | 通常 ≤ 2% 下降 |
| 混合（INT8权重、float激活） | ~50 KB | 中等 | ~1% 下降 |

**选择: Full INT8量化**。ESP32S3没有针对float32推理优化的FPU，因此INT8既更小又更快。使用200个训练样本作为代表性数据集进行校准。预期 ≤ 2% 的精度下降在NFR-7的范围内可接受。

### 3.5 下采样方法（ESP32端）

| 选项 | 实现复杂度 | 质量 |
|:----|:--------:|:-----|
| 最近邻 | 非常简单 | 锯齿伪影，输出噪声大 |
| 双线性插值 | 中等（每像素浮点运算） | 平滑但计算量更大 |
| **5×5区域平均** | 简单（整数累加） | 内置抗锯齿；降噪效果 |

**选择: 5×5区域平均**。从320×240中心裁剪240×240，然后每个5×5块平均为一个像素，恰好得到48×48。这与选定的分辨率自然匹配，仅需整数运算，并且内在地减少噪声——起到低通滤波器的作用。

### 3.6 汇总表

| 设计维度 | 选择 | 被否决的替代方案 | 核心理由 |
|:--------|:-----|:-------------|:---------|
| 输入分辨率 | 48×48 | 32×32、64×64 | 细节与SRAM预算的平衡 |
| 颜色通道 | 灰度 | RGB | 身份识别基于结构而非颜色；3倍计算节省 |
| 模型深度 | 3层Conv | 2层、4层、MobileNet | 符合资源预算；复用已验证的算子集 |
| 量化 | Full INT8 | Float32、混合 | ESP32S3上更小更快；≤ 2% 精度代价 |
| 下采样 | 5×5区域平均 | 最近邻、双线性 | 与240→48的自然匹配；内置抗锯齿 |

## 4. 机器学习流水线

### 4.1 预处理流程

**Python（训练端）**:
```
PNG图像（320×240或任意尺寸）
    → 转换为灰度（PIL .convert('L')）
    → 缩放到48×48（双线性插值）
    → 归一化: pixel / 255.0
    → Z-score: (x - IMAGE_MEAN) / IMAGE_STD
    → 输出形状: (48, 48, 1) float32
```

**ESP32（推理端）**:
```
320×240 RGB565 原始帧
    → 中心裁剪 240×240（跳过左右各40像素）
    → 5×5区域平均下采样 → 48×48
    → RGB565→灰度: Y = (5R + 9G + 2B) >> 4
    → 归一化: (Y / 255.0 - IMAGE_MEAN) / IMAGE_STD
    → 输出: float[48×48]
```

**对齐要求**: 两端流水线对同一输入图像必须产生数值等价的输出。通过 `test_case.h` 验证（见第7节）。

### 4.2 模型架构

**框架**: TensorFlow 2.x / Keras → TFLite INT8 → TFLite Micro

```
Input (48, 48, 1)
    │
Conv2D(8, 3×3, relu)    → (46, 46, 8)
MaxPooling2D(2×2)        → (23, 23, 8)
Dropout(0.1)
    │
Conv2D(16, 3×3, relu)   → (21, 21, 16)
MaxPooling2D(2×2)        → (10, 10, 16)
Dropout(0.1)
    │
Conv2D(32, 3×3, relu)   → (8, 8, 32)
MaxPooling2D(2×2)        → (4, 4, 32)
Dropout(0.1)
    │
Flatten                  → (512)
Dense(32, relu)          → (32)
Dense(4, softmax)        → (4)  ← 成员A / 成员B / 成员C / Unknown

总参数量: ~22,400
TFLite INT8 模型大小: ~40 KB
```

**算子解析器**: `Conv2D`、`MaxPool2D`、`Flatten`、`FullyConnected`、`Softmax` — 与keywords项目完全相同的算子集。无需新增TFLite算子。

### 4.3 训练配置

| 参数 | 值 |
|:----|:---|
| 优化器 | Adam（lr = 0.001） |
| 损失函数 | sparse_categorical_crossentropy |
| 轮数 | 100，带EarlyStopping（patience = 20, monitor = val_loss） |
| 批大小 | 32 |
| 数据划分 | 70% 训练 / 15% 验证 / 15% 测试 |
| 数据增强 | 水平翻转、±15°旋转、亮度[0.7, 1.3]、缩放0.1 |

### 4.4 量化与导出

- 使用代表性数据集（200个训练样本）进行Full INT8量化
- 输入/输出张量均为INT8
- 导出: 通过 `write_model_c_file()` 工具生成 `model.c` + `model.h`（复用keywords项目）
- 验证: 在测试集上评估TFLite INT8模型，与浮点基线对比精度

## 5. ESP32固件设计

### 5.1 文件结构

```
face/esp32/main/
├── main.cpp          # 应用主循环: 摄像头 → 预处理 → 推理 → LED → 串口
├── camera.h/.cpp     # 从camera/esp32/main/直接复制（不修改）
├── preprocess.h/.cpp # 新建: RGB565 → 48×48灰度 → 归一化
├── inference.h/.cpp  # 从keywords/esp32/main/改编（2D输入、4类、80 KB arena）
├── model.h/.c        # 由python/main.py生成
├── test_case.h       # 由python/generate_test_case.py生成
└── test.h/.cpp       # 从keywords/esp32/main/改编（图像测试向量）
```

### 5.2 串口协议

每个推理周期发送两个数据包:

```
[帧数据包]
  "===FRAME===\n"               ← 与camera项目相同的前导码
  <320×240×2字节 RGB565>        ← 153,600字节

[结果数据包]
  "===RESULT===\n"              ← 新增的结果前导码
  <1字节: uint8 类别编号>
  <16字节: 4×float32 分数>      ← 小端序
```

Python解析: `cls, *scores = struct.unpack('<Bffff', data[1:18])`

### 5.3 内存布局

| 缓冲区 | 大小 | 放置位置 |
|:------|-----:|:--------|
| `image_buffer[320×240×2]` | 153,600 B | PSRAM (SPIRAM) |
| `face_features[48×48]` | 9,216 B | SRAM |
| `tensor_arena[80×1024]` | 81,920 B | SRAM |
| `prediction[4]` | 16 B | SRAM |
| **SRAM总计** | **约91 KB** | 在512 KB预算内 |

### 5.4 LED逻辑

```c
if (max_score >= 0.80f && predicted_class < 3)
    gpio_set_level(LED_PIN, 0);   // 点亮（低电平有效）= 允许通行
else
    gpio_set_level(LED_PIN, 1);   // 熄灭 = 拒绝通行
```

## 8. 进阶功能

### 进阶1: 口罩/眼镜属性检测
- 第二个小型CNN（同样48×48灰度输入）: 3类输出（无 / 口罩 / 眼镜）
- 模型大小: ~5 KB；独立的32 KB tensor arena
- 两个模型共享 `face_features[]` — 在主循环中顺序运行
- Python叠加层添加属性标签

### 进阶2: 门禁场景化
- 仅修改Python端（无需更新固件）
- 去抖: 连续5帧中3帧一致且置信度 > 0.80 时触发访问事件
- 动画 "允许通行 / 拒绝通行" 面板
- 滚动访问日志写入 `access_log.csv`

### 进阶3: 活体检测
- 仅Python端；无需新模型
- 最近8帧的滚动缓冲区（numpy数组）
- `is_live = np.mean(np.var(frame_stack, axis=0)[center_crop]) > THRESHOLD`
- 门禁条件: `confidence > 0.80` 且 `is_live == True` 同时满足
